# discuss 技能质量评估报告

**评估对象**：discuss-skill（v1.1.0，2025-06-27）
**评估依据**：WorkBuddy skill-creator 官方规范 + 实际可用性审查
**评估日期**：2026-06-28

---

## 一、总体评价

| 维度 | 评分（/10） | 简评 |
|------|------------|------|
| 元数据规范 | 5 | 缺关键字段，描述风格不规范 |
| 结构组织 | 7.5 | 章节清晰，但篇幅偏长 |
| 触发准确性 | 6 | 触发词泛化，易误触发 |
| 流程清晰度 | 7 | 主流程清楚，决策点模糊 |
| 实现细节 | 5 | 持久化检测、云端 API 调用均缺落地 |
| 示例与模板 | 8 | 三种模式示例完整，模板可用 |
| 边界处理 | 4 | 错误恢复、回退、并发场景缺位 |
| 资源组织 | 6 | 目录命名偏离官方约定 |

**综合评分：6.6 / 10** —— 一个"能用、思路清楚，但工程化不足"的技能。框架设计有价值，但落地执行的确定性不够。

## 二、优势

### 1. 框架选型扎实
- 基于**金字塔原理 + SCQA**，是经过验证的结构化思考方法
- 议题拆解遵循 **MECE** 原则，避免重叠和遗漏
- 三层状态机（❌ → ⏳ → ✅）让进度可视化

### 2. 持久化三模式设计有前瞻性
- **本地 / 对话 / 云端**三档覆盖了从无文件能力到企业云端协同的完整场景
- 默认对话模式是合理选择（兜底最稳）
- 云端模式支持飞书、语雀等主流平台，思路正确

### 3. 递归拆解 + 深度追问的混合策略
- 用"可继续拆解？"判断走拆解还是追问，避免无限递归
- 深度追问引入 `verdict`（solid / needs-work / invalid）机制，强制对论据做质量裁决，这是亮点

### 4. 示例和模板配套完整
- 3 个示例覆盖三种持久化模式，每种都有完整对话流
- 2 个模板（notes / summary）字段齐全，可直接套用
- summary 模板还包含"落地规划""风险与应对"，超出基础 SCQA，更贴近实际使用

### 5. 中断恢复机制有考虑
- 支持切换议题、查看详情、返回清单
- 强制保存规则明确：状态变更前必须先持久化

## 三、劣势

### A. 元数据层（影响：技能能否被正确识别和后续维护）

1. **缺 `agent_created: true` 字段**
   - 官方规范明确要求：YAML frontmatter 必须包含此字段，否则 `skill_manage` 无法修改/删除该技能
   - 当前 frontmatter 只有 `name` 和 `description`，不合规

2. **description 用了"Use when..."句式**
   - 官方建议用第三人称："This skill should be used when..."
   - 当前："Use when analyzing complex problems..." 偏口语化

3. **description 没体现差异化能力**
   - 没提到"递归拆解""三种持久化模式""verdict 裁决机制"这些核心卖点
   - 只说"SCQA framework"，和市面上其他 SCQA 工具无区分度

### B. 触发层（影响：会不会误触发或漏触发）

4. **触发词过于泛化**
   - `分析这个话题` / `analyze` 这类词在日常对话中太常见
   - 用户随便说"分析一下这个数据"也可能触发，造成误判
   - 应该用更明确的 `/discuss` 或 `讨论` 作为主触发，弱化 `analyze`

5. **`/discuss` 暗示是 slash command，但实际不是**
   - 这只是个 skill，没有注册 slash command
   - 用户输入 `/discuss` 不会得到特殊处理，容易误导

### C. 流程层（影响：执行时会不会卡住或跑偏）

6. **"补充上下文"作为强制第一步可能惹恼用户**
   - 流程第 1 步固定要用户说明"为什么讨论这个话题"
   - 但很多场景用户就是想直接开干，强制问背景反而拖慢节奏
   - 应改为：可选步骤，或仅在信息明显不足时才追问

7. **"可继续拆解"的判断标准太主观**
   - 原文："议题复杂、多维度、需进一步明确细节"
   - 没有可操作的判定规则，全靠模型临场发挥
   - 建议给出量化信号，例如：议题包含≥2个独立子问题 / 当前信息不足以直接给出 verdict / 用户明确要求细化

8. **深度追问的 verdict 机制示例不足**
   - 只展示了 `solid` 一种 verdict
   - `needs-work` 和 `invalid` 长什么样？模型遇到这两种该怎么处理？要不要回退到拆解？都没说

9. **递归深度"3-4 层为宜"是软约束**
   - 没有硬性上限，模型可能拆到 6-7 层把上下文撑爆
   - 应明确："超过 4 层必须停止拆解，转为深度追问"

### D. 实现层（影响：能不能真的跑起来）

10. **持久化模式检测逻辑是黑箱**
    - "检测到本地文件能力"——具体检测什么？检查哪些工具？
    - "检测到语雀 API"——怎么检测？环境变量？MCP 连接器？
    - 应明确：检查当前会话是否有 `mkdir`/`write`/`edit` 工具，或检查是否有 `lark-doc`/`yuque` 等 connector

11. **云端模式完全没给 API 调用细节**
    - 只说"在指定地址下创建文档"，但用什么工具创建？
    - 飞书应该走 `lark-doc` skill / `lark-drive` skill
    - 语雀的 API 端点、鉴权方式只字未提
    - 这是最大的实现空洞——写得很美，跑不起来

12. **每次状态变更都强制持久化，token 开销大**
    - 长讨论里可能产生几十次文件读写
    - 没有"批量持久化"或"关键节点持久化"策略
    - 建议区分：状态变更（轻量，内存跟踪）vs 阶段性结论（重量，必须落盘）

### E. 边界处理层（影响：异常场景会不会崩）

13. **用户输入非法议题号怎么办？** 没说
14. **云端文档创建失败怎么办？** 没说（回退到本地？还是对话模式？）
15. **讨论中途新信息推翻了已有结论怎么办？** 没说（要不要重新拆解？）
16. **用户想修改之前标 ✅ 的议题怎么办？** 没说（状态能回退吗？）
17. **上下文超长时如何压缩历史？** 没说

### F. 资源组织层（影响：是否符合官方约定）

18. **目录命名偏离官方规范**
    - 官方约定：`scripts/` / `references/` / `assets/`
    - 当前用的是：`examples/` / `templates/`
    - `templates/` 里的 notes-template.md、summary-template.md 实际是 **assets**（用于输出的模板文件），应改名 `assets/`
    - `examples/` 里的示例文档是 **references**（供模型参考的文档），应改名 `references/`

19. **SKILL.md 篇幅偏长**
    - 官方建议 <5k words，当前约 8KB，偏长
    - 示例对话、模式说明这些可以挪到 `references/`，SKILL.md 只保留核心流程

### G. 内容一致性

20. **SKILL.md 和 summary-template 结构不一致**
    - SKILL.md 里的 SCQA 文档只有 S/C/Q/A 四段
    - summary-template.md 里多了"议题分析汇总""落地规划""风险与应对"
    - 模型执行时会困惑：到底按哪个生成？

21. **版本日期穿越**
    - 标注更新日期 2025-06-27，但实际今天是 2026-06-28
    - 要么是写错了年份，要么是技能已一年未维护

## 四、修改建议（按优先级排序）

### P0 - 必须修（影响合规和基础可用性）

| # | 问题 | 修改 |
|---|------|------|
| 1 | 缺 `agent_created: true` | frontmatter 加上此字段 |
| 2 | 云端模式无 API 落地 | 至少为飞书模式补充：调用 `lark-doc` skill 创建文档；语雀若无 connector 则标注"需用户手动提供 API token"或暂时移除 |
| 3 | 持久化检测逻辑黑箱 | 改为："检测当前会话可用工具——若含 write/edit 则用本地模式；若含 lark-* connector 且用户提供飞书链接则用云端模式；否则对话模式" |
| 4 | SKILL.md 与模板结构不一致 | 统一为 summary-template.md 的完整结构（含落地规划、风险），SKILL.md 里的简化版删掉 |

### P1 - 应该修（影响执行确定性和体验）

| # | 问题 | 修改 |
|---|------|------|
| 5 | 触发词过泛 | 主触发词只保留 `/discuss` 和 `讨论`；`analyze` 降级为弱触发或移除 |
| 6 | "补充上下文"强制第一步 | 改为：默认跳过，仅在根问题信息不足以拆解时才追问背景 |
| 7 | 拆解停止标准模糊 | 给出硬规则："议题可拆解为≥2个独立子问题且当前无直接结论 → 继续拆解；否则进入追问；递归深度≥4 强制停止" |
| 8 | verdict 缺 needs-work/invalid 示例 | 补 2 个反例，并说明遇到这两种时：needs-work→补充论据后重判；invalid→回退到拆解或与用户确认方向 |
| 9 | 目录改名 | `examples/` → `references/`；`templates/` → `assets/` |
| 10 | 持久化开销大 | 区分"状态跟踪"（内存即可）和"阶段性结论"（必须落盘），减少文件读写频次 |

### P2 - 可以优化（提升健壮性和可维护性）

| # | 问题 | 修改 |
|---|------|------|
| 11 | 缺错误恢复指引 | 增加"异常处理"章节：非法输入、云端失败、结论推翻各自的应对 |
| 12 | description 不够差异化 | 改为："This skill should be used when analyzing complex problems, making decisions, or planning strategies. It recursively decomposes topics via MECE, validates arguments through deep questioning with verdict ratings, and produces SCQA documents with three persistence modes (local/chat/cloud)." |
| 13 | SKILL.md 偏长 | 把"使用示例"整段挪到 `references/usage-examples.md`，SKILL.md 只留流程骨架 |
| 14 | 缺反模式说明 | 增加"不要做"清单：不要一次拆超过 5 个子议题、不要在追问阶段又拆解、不要跳过 verdict 直接下结论 |
| 15 | 版本日期校准 | 更新为当前真实日期，或引入语义化版本 + changelog |

## 五、一句话总结

> **框架思路 8 分，工程落地 5 分。** 这个技能最该补的不是更多功能，而是把"怎么检测、怎么调用、出错怎么办"这三件事写清楚—— otherwise 模型每次执行都得临场发挥，确定性不够。
