# Discuss — 结构化议题研讨

基于金字塔原理（Barbara Minto），将复杂议题逐层拆解为子议题，通过深度追问和脑暴交替推进，最终汇总为完整 SCQA 文档与落地规划。议题拆解、深度追问、verdict 裁决均已内置，无需依赖其他技能。

**版本**：v2.1.0 | **更新日期**：2026-06-28

## When to Use

本技能为通用 skill，可运行于支持 skill 机制的任意智能体平台。触发方式因平台而异：

- **关键词触发**：用户消息含 `讨论` / `讨论下` / `/discuss` / `discuss this` 等
- **语义触发**：平台根据 description 语义匹配（如"分析""拆解"复杂话题）

**适用场景**：
- 复杂问题需要结构化分析
- 团队讨论、决策制定、战略规划
- 需要生成完整解决方案文档
- 需要明确执行步骤和时间节点

**不适用**（避免误触发）：
- 单点事实查询（直接回答即可）
- 纯代码调试（用 debugging 类技能）
- 只需列清单的简单任务

## Core Pattern

```
根问题 → 拆解为 N 个子议题 → 每个子议题判断：
  ├─ 可继续拆解？(见判定规则) → 是 → 递归拆解
  └─ 否 → 深度追问 → 给出 verdict → 获得论据
所有议题 ✅ → 汇总成完整 SCQA 文档
```

### 拆解停止规则（硬性，避免无限递归）

判定**继续拆解**需同时满足：
1. 议题可拆为 ≥2 个相互独立的子问题
2. 当前信息不足以直接给出 verdict
3. 递归深度 < 4 层

判定**进入深度追问**（满足任一）：
- 议题已具体到可验证的假设
- 递归深度已达 4 层（强制停止拆解）
- 用户明确要求"直接问"

**递归深度上限：4 层。超过必须停止拆解，转入深度追问。** 议题号层级即深度计数——`1` 是 1 层，`1.2` 是 2 层，`1.2.3.4` 是 4 层上限。

### MECE 检查

每次拆解后必须检查子议题是否满足 MECE：
- **相互独立**：子议题之间无重叠
- **完全穷尽**：子议题合起来覆盖父议题全部范围

不满足时重新拆解。子议题数量通常 3-5 个，最多不超过 5 个。

拆解示例（含正例和反例）见 `references/examples.md` 第二节。

## Process

```
1. 定义根问题：明确核心问题（必要时追问 1-2 个澄清问题）
2. [可选] 补充上下文：仅当根问题信息不足以拆解时，追问背景与动机
3. 检测持久化模式（见 Persistence Detection）
4. 拆解议题：按 MECE 生成 3-5 个子议题，标记 ❌
5. 展示清单：列出所有议题及状态
6. 用户选择议题 → 标记 ⏳
7. 讨论议题：递归拆解 或 深度追问 → 给出 verdict → 记录论据
8. 循环 6-7 直至所有叶子议题 ✅
9. SCQA 汇总：生成文档（结构见 assets/summary-template.md）
10. 落地规划：执行步骤、执行节奏、风险与应对
```

**关键：上下文追问是可选的。** 默认直接进入拆解；只有当根问题过于笼统（如"分析项目情况"）才追问背景。不要为每个议题强制问"为什么讨论这个"。

**持久化时机**：本地/云端模式下，步骤 4-10 每次产生新状态或论据时持久化（用 edit 局部更新）。对话模式跳过持久化，仅在对话中展示。

## Persistence Detection

按以下优先级检测当前会话可用的持久化方式。**用户显式指定时，以用户指定为准。**

| 优先级 | 条件 | 模式 |
|--------|------|------|
| 1 | 用户消息明确要求本地保存（含"本地""存文件"等意图） | 本地模式 |
| 2 | 用户消息含云文档链接 + 对应 connector 已连接 | 云端模式 |
| 3 | 用户消息明确要求纯对话（含"不存""纯对话"等意图） | 对话模式 |
| 4 | 以上均不满足 | 对话模式（默认兜底） |

**默认：对话模式。** 未明确指定时，不主动创建文件，避免对用户工作区产生副作用。用户可随时说"存到本地"切换到本地模式。

### 本地模式

用户明确要求本地保存，或会话中途切换到本地时启用。

文件结构：
```
docs/discuss/YYYY-MM-DD-[主题]/
├── notes.md      # 讨论过程记录（参照 assets/notes-template.md）
└── summary.md    # 最终 SCQA 汇总（参照 assets/summary-template.md）
```

操作：用 `mkdir -p` 建目录，`write` 建文件，`edit` 更新。状态变更时用 `edit` 追加或更新对应段落。

### 对话模式（默认）

未明确指定时默认启用。不进行任何文件操作。所有内容在对话中展示。每次展示议题清单时附注 `💾 对话模式，内容仅在会话内有效，可随时说"存到本地"切换`。

### 云端模式

**前置条件**：用户提供了云文档链接，且对应 connector 已连接。

支持的云文档平台与对应 skill：

| 平台 | connector | 操作 skill |
|------|-----------|------------|
| 飞书 | `feishu` | `lark-doc` |
| 其他（钉钉/腾讯文档/语雀等） | 对应 connector | 对应 doc skill |

**工作流程**：
1. 识别链接对应的平台，加载相应 doc skill 上下文
2. 从链接提取文档 token
3. 通过该 skill 在父文档下创建子文档：
   - `<议题名>-讨论过程`（对应 notes）
   - `<议题名>-SCQA`（对应 summary）
4. 后续每次持久化时，用该 skill 更新对应文档内容
5. 展示状态时附文档链接

**若 connector 未连接**：告知用户需先在连接器管理中启用对应平台，或回退到本地/对话模式（询问用户）。

## Verdict System

深度追问后必须给出 verdict，对论据质量做明确裁决：

| verdict | 含义 | 处理方式 |
|---------|------|----------|
| `solid` | 论证充分，逻辑自洽 | 标记议题 ✅，记录论据 |
| `needs-work` | 有漏洞，论据不足 | 补充追问 1-2 轮；若仍不足，标注缺失点后标记 ✅（带 caveat） |
| `invalid` | 前提不成立，方向错误 | 与用户确认是否回退到拆解阶段重新拆，或调整根问题 |

**禁止跳过 verdict 直接下结论。** 即使是显然成立的议题，也要明确标注 `solid` 并附 1 句理由。

详细示例（完整讨论流程 + 三模式持久化差异 + verdict 三种裁决）见 `references/examples.md`。

## Error Handling

| 异常场景 | 处理方式 |
|----------|----------|
| 用户输入非法议题号 | 提示当前有效序号范围，请用户重新选择 |
| 云端文档创建/更新失败 | 重试 1 次；仍失败则询问是否回退到本地模式，已生成内容不丢失 |
| 讨论中新信息推翻已有结论 | 将相关议题状态从 ✅ 回退到 ⏳，重新讨论；在 notes 中记录"结论修订"事件 |
| 用户想修改已 ✅ 的议题 | 允许回退状态为 ⏳，重新讨论；旧结论保留在 notes 历史段 |
| 上下文过长 | 压缩已 ✅ 议题的细节，只保留结论；完整细节已在 notes 中持久化 |
| 用户中途想结束 | 询问是否生成当前进度的 SCQA 文档（部分议题未完成也生成，标注未完成项） |

## Anti-patterns（不要做）

- ❌ 一次拆解超过 5 个子议题（信息过载）
- ❌ 在深度追问阶段又发起拆解（混用两种模式）
- ❌ 跳过 verdict 直接标记 ✅
- ❌ 每次状态变更都全量重写 notes.md（用 `edit` 局部更新，减少 token 开销）
- ❌ 强制追问每个议题的"背景动机"（仅根问题信息不足时才问）
- ❌ 把 `/discuss` 当必然触发的 slash command（触发依赖平台机制，可能是关键词也可能是语义匹配）

## Topic State & Navigation

### 状态流转

```
❌ 未讨论 → ⏳ 进行中 → ✅ 已解决
            ↑    │
            └────┘ (结论被推翻 / 用户要求重做)
```

**持久化与状态跟踪的区别**：
- **持久化**：将状态和论据写入文件（本地模式写 notes.md，云端模式写云文档）。仅本地/云端模式涉及。
- **状态跟踪**：在对话中展示当前议题状态（所有模式都做）。对话模式只有状态跟踪，无持久化。

本地/云端模式下，状态变更先持久化再展示。允许从 ✅ 回退到 ⏳（结论被推翻或用户要求重做时），旧结论保留在 notes 历史段。

### 议题导航操作

| 用户操作 | 说明 |
|----------|------|
| 输入序号（如 `1`、`1.2`） | 讨论指定议题 |
| `切换到 X` | 中断当前议题，切换到 X（先保存当前进度） |
| `查看 X` | 查看议题 X 的详情 |
| `返回清单` | 返回议题清单视图 |
| `结束讨论` | 结束并生成 SCQA 文档 |
