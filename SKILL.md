# Discuss — 结构化议题研讨

基于金字塔原理（Barbara Minto），将复杂议题逐层拆解为子议题，通过深度追问和脑暴交替推进，最终汇总为完整 SCQA 文档与落地规划。

**自包含技能**：无需依赖任何外部技能，所有功能内置。

## When to Use

**触发词**：讨论、讨论下、分析这个话题、/discuss、analyze

**适用场景**：
- 复杂问题需要结构化分析
- 需要生成完整的解决方案文档
- 团队讨论和决策制定
- 战略规划和方案设计
- 需要明确执行步骤和时间节点

## Core Pattern

### 递归拆解流程

```
根问题 (Root Problem)
    │
    ▼
┌─────────────────────────────────────┐
│ 议题拆解：拆解为N个子议题           │
└─────────────────────────────────────┘
    │
    ├─→ 子议题1 ──→ 可继续拆解？──→ 是 ──→ 递归拆解
    │                              └→ 否 ──→ 深度追问 ──→ 获得论据
    │
    ├─→ 子议题2 ──→ 可继续拆解？──→ 是 ──→ 递归拆解
    │                              └→ 否 ──→ 深度追问 ──→ 获得论据
    │
    └─→ 子议题N ──→ ...（同上）
    │
    ▼
所有议题解决 ──→ 汇总成完整SCQA文档
```

### 判断标准

**可继续拆解的条件：**
- 议题仍然复杂，包含多个维度
- 存在不同视角需要分析
- 需要进一步明确细节

**进入深度追问的条件：**
- 议题已经足够具体
- 需要深入挖掘根因
- 需要获得论据支持
- 需要验证假设

## 议题拆解（内置）

当需要拆解议题时，遵循以下步骤：

1. **理解根问题**：明确要讨论的核心问题是什么
2. **识别维度**：从不同角度分析问题（如：原因、影响、解决方案）
3. **MECE 检查**：确保子议题相互独立、完全穷尽
4. **生成子议题**：通常 3-5 个子议题
5. **标记状态**：所有子议题初始状态为 ❌

**拆解示例**：
```
根问题：为什么最近项目延期严重？
├─ 1. 资源问题
│  ├─ 1.1 人员不足
│  └─ 1.2 技能不匹配
├─ 2. 流程问题
│  ├─ 2.1 需求变更频繁
│  └─ 2.2 评审环节多
└─ 3. 技术问题
   └─ 3.1 技术债务
```

## 深度追问（内置）

当需要深度追问时，遵循以下步骤：

1. **无情盘问**：对每个假设、前提、数据依赖逐一质疑
2. **逐分支遍历**：
   - 逻辑前提是否成立？
   - 数据来源是否可靠？
   - 因果关系是否充分？
   - 是否有遗漏的替代解释？
   - 边界条件是否考虑？
   - 实施可行性如何？
3. **逐个决策**：每个质疑点给出推荐答案
4. **给出 verdict**：
   - `solid`：论证充分，逻辑自洽
   - `needs-work`：有漏洞，需要修正
   - `invalid`：前提不成立，方向错误

**追问示例**：
```
议题：人员不足
质疑：当前团队规模8人，项目需求12人
verdict：solid（数据支撑充分）

质疑：离职率20%是否准确？
verdict：needs-work（需要核实数据来源）
```

## 流程步骤

```
1. 定义问题 (Define)
   └─ 明确根问题
   ↓
2. 检测持久化模式 (Detect)
   └─ 自动检测可用的持久化方式
   ↓
3. 拆解议题 (Decompose)
   ├─ 内置拆解流程，生成N个子议题
   └─ 【持久化】保存初始议题清单
   ↓
4. 展示议题清单 (Display)
   └─ 清晰列出所有子议题及状态
   ↓
5. 用户选择议题 (Select)
   ├─ 用户选择要讨论的议题（可随时切换）
   └─ 【持久化】更新议题状态为 ⏳
   ↓
6. 讨论当前议题 (Discuss)
   ├─ 可继续拆解？──→ 是 ──→ 递归拆解
   │                  └─ 【持久化】添加子议题
   │                  └→ 否 ──→ 深度追问
   │                             └─ 【持久化】记录论据
   ├─ 标记议题状态
   └─ 【持久化】更新状态为 ✅
   ↓
7. 循环直至所有议题解决
   ↓
8. SCQA 汇总 (Synthesize)
   ├─ Situation (背景)
   ├─ Complication (冲突)
   ├─ Question (问题)
   ├─ Answer (解决方案)
   └─ 【持久化】保存汇总文档
   ↓
9. 落地规划 (Plan)
   ├─ 制定执行节奏和检查点
   └─ 【持久化】更新汇总文档
```

## 持久化机制

### 自动检测

技能启动时自动检测可用的持久化方式：

1. **检测本地文件能力**：尝试执行 `mkdir -p /tmp/discuss-test`
   - 成功 → 使用本地文件模式
   - 失败 → 继续检测

2. **询问用户偏好**：
   - 本地文件（推荐）
   - 语雀文档
   - 飞书文档
   - 纯对话模式

### 模式选择

用户可以手动指定模式：
- `本地文件` / `local` / `file`
- `语雀` / `yuque`
- `飞书` / `feishu`
- `纯对话` / `chat` / `pure`

### 降级方案

如果当前模式失败：
1. 自动降级到下一个可用模式
2. 提示用户当前模式已变更
3. 尝试迁移已有数据到新模式

**场景**：分析为什么最近项目延期严重

```

=== Step 1: 定义问题 ===
根问题：项目延期问题分析

=== Step 2: 检测持久化模式 ===
检测到本地文件能力，使用本地文件模式

=== Step 3: 拆解议题 ===
内置拆解流程，生成3个议题
创建目录：docs/discuss/2025-06-26-项目延期分析/
创建文件：notes.md

=== Step 4: 展示议题清单 ===
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤

│ ❌ 3. 技术问题                                              │
├─────────────────────────────────────────────────────────────┤

└─────────────────────────────────────────────────────────────┘

请选择要讨论的议题：1

=== Step 5: 讨论议题1 ===
用户选择：1. 资源问题
内置拆解：
├─ 1.1 人员不足
└─ 1.2 技能不匹配
```

## 注意事项

- **递归深度控制**：避免过度拆解，一般3-4层为宜
- **议题独立性**：子议题之间尽量减少重叠
- **论据充分性**：深度追问要获得足够论据支持
- **状态跟踪**：实时更新每个议题的解决状态
- **及时汇总**：所有议题解决后立即汇总文档
- **灵活导航**：允许用户随时切换议题，支持中断和恢复
- **强制保存**：每次状态变更必须立即持久化，防止进度丢失

## 议题导航规则

### 用户操作

| 操作 | 说明 |
|------|------|
| 输入序号 | 讨论指定议题（如：1、2.1） |
| `切换到 X` | 中断当前议题，切换到议题X |
| `查看 X` | 查看议题X的详情 |
| `返回清单` | 返回议题清单 |
| `结束讨论` | 结束讨论，汇总文档 |

### 中断处理

当用户中断当前议题时：
1. **立即保存**当前讨论进度（包括未完成的论据）
2. 标记议题状态为进行中（⏳）
3. 切换到用户指定的议题
4. 新议题开始讨论

**强制保存规则**：
- 任何导致议题状态变更的操作都必须先保存
- 切换议题前必须保存当前议题的进度
- 退出 discuss 前必须保存所有未保存的进度

### 议题状态流转

```
❌未讨论 → ⏳进行中 → ✅已解决
            ↑    │
            └────┘ (中断后恢复)
```

## SCQA 框架详解

| 元素 | 说明 | 输出要求 |
|------|------|----------|
| **S (Situation)** | 当前背景和现状 | 客观描述，用数据支撑 |
| **C (Complication)** | 遇到的冲突和挑战 | 明确痛点和障碍 |
| **Q (Question)** | 核心要解决的问题 | 清晰、可回答的问题 |
| **A (Answer)** | 解决方案 | 可执行的行动步骤 |
