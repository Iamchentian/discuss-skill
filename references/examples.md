# 示例

统一用"项目延期"场景。包含：讨论流程（三模式通用）、拆解示例、三模式持久化差异、verdict 三种裁决。

---

## 一、讨论流程（三模式通用）

以下流程对所有持久化模式相同，差异只在持久化动作（见第三节）。

```
用户：/discuss 分析为什么最近项目延期严重

技能：根问题：项目延期问题分析

技能：按 MECE 拆解议题：
├─ 1. 资源问题
├─ 2. 流程问题
└─ 3. 技术问题

技能：[持久化：写入议题清单]

┌─────────────────────────────────────────────────────────────┐
│ 议题清单                                                     │
├─────────────────────────────────────────────────────────────┤
│ ❌ 1. 资源问题                                              │
│ ❌ 2. 流程问题                                              │
│ ❌ 3. 技术问题                                              │
├─────────────────────────────────────────────────────────────┤
│ 进度：0/3                                                   │
└─────────────────────────────────────────────────────────────┘

用户：1

技能：议题 1 状态 → ⏳。[持久化]
技能：拆解资源问题：
├─ 1.1 人员不足
└─ 1.2 技能不匹配

用户：1.1

技能：议题 1.1 状态 → ⏳。[持久化]
技能：1.1 已具体到可验证假设，进入深度追问：
- 当前团队规模：8 人
- 项目需求：12 人（依据：项目计划书人力需求表）
- 离职率：20%（依据：HR 季度报告）
- 招聘周期：3 个月（依据：近半年招聘数据）

verdict：solid
理由：数据来源明确，因果链清晰——8 人 < 12 人需求，缺口 4 人且短期难补齐。

技能：议题 1.1 状态 → ✅，记录论据。[持久化]

（循环直至所有叶子议题 ✅）

技能：汇总 SCQA 文档（结构见 assets/summary-template.md）。[持久化]
```

## 二、拆解示例

### 正例：符合 MECE

根问题"项目延期"拆为 3 个子议题：
- 1. 资源问题（人/设备）
- 2. 流程问题（协作/管理）
- 3. 技术问题（架构/债务）

**MECE 检查**：
- 相互独立：三类原因无重叠——资源是"有没有"，流程是"怎么协作"，技术是"代码质量"
- 完全穷尽：延期原因无非人、流程、技术三方面，覆盖全
- ✅ 通过

继续拆解"1. 资源问题"：
- 1.1 人员不足（数量缺口）
- 1.2 技能不匹配（质量缺口）

**MECE 检查**：
- 相互独立：数量和质量是两个独立维度
- 完全穷尽：资源问题要么数量不够，要么质量不行
- ✅ 通过

### 反例：违反 MECE

❌ **重叠**（不满足"相互独立"）：
```
1. 人员不足
2. 招聘困难
3. 技能不匹配
```
问题：1 和 2 重叠——"人员不足"的因往往就是"招聘困难"。应合并为"人员数量问题"，或拆成"数量"和"质量"两个正交维度。

❌ **遗漏**（不满足"完全穷尽"）：
```
1. 人员不足
2. 技术债务
```
问题：遗漏了"流程/协作"维度。延期可能既不是人不够也不是技术差，而是跨团队协作卡点。

### 拆解深度控制

```
1. 资源问题              ← 第 1 层
├─ 1.1 人员不足          ← 第 2 层
│  ├─ 1.1.1 招聘周期长   ← 第 3 层
│  └─ 1.1.2 离职率高     ← 第 3 层
└─ 1.2 技能不匹配        ← 第 2 层
```

达到 4 层（如 `1.1.1.1`）必须停止拆解，转入深度追问。议题号段数即深度计数。

## 三、三模式持久化差异

讨论流程完全相同，差异只在每次 `[持久化]` 动作：

### 本地模式

检测条件：用户明确要求本地保存（含"本地""存文件"等意图），或会话中途从对话模式切换。

```
首次：mkdir -p docs/discuss/2026-06-28-项目延期/
      write notes.md（参照 assets/notes-template.md 初始化）

每次状态变更：edit notes.md（局部更新议题状态、追加论据）

最终：write summary.md（参照 assets/summary-template.md）

展示：💾 内容已保存至 docs/discuss/2026-06-28-项目延期/notes.md
```

notes.md 片段：
```markdown
| 序号 | 议题 | 状态 | 论据来源 |
|------|------|------|----------|
| 1 | 资源问题 | ⏳ | - |
| 1.1 | 人员不足 | ✅ | 内置追问 |
| 1.2 | 技能不匹配 | ❌ | - |
```

### 对话模式（默认）

检测条件：用户未明确指定持久化方式时的默认兜底。无文件操作。

```
每次状态变更：在回复中展示新状态，附注
  "💾 对话模式，内容仅在会话内有效，可随时说'存到本地'切换"

最终：SCQA 文档直接输出在对话中
```

### 云端模式

检测条件：用户提供了云文档链接 + 对应平台 connector 已连接。

| 平台 | connector | 操作 skill |
|------|-----------|------------|
| 飞书 | `feishu` | `lark-doc` |
| 其他（钉钉/腾讯文档/语雀等） | 对应 connector | 对应 doc skill |

工作流程（以飞书为例）：
```
首次：加载 lark-doc skill 上下文
      从链接提取 doc token
      通过 lark-doc 在父文档下创建子文档：
        - 项目延期分析-讨论过程（对应 notes）
        - 项目延期分析-SCQA（对应 summary）

每次状态变更：通过对应 skill 更新"讨论过程"文档

最终：通过对应 skill 更新"SCQA"文档

展示：💾 内容已保存至云文档 [项目延期分析-讨论过程](链接)
```

connector 未连接时：告知用户需先启用对应平台 connector，或回退本地/对话模式（询问）。

## 四、Verdict 三种裁决

深度追问后必须给 verdict。以下是三种裁决的完整示例。

### solid — 论证充分

```
议题：1.1 人员不足
论据：8 人 < 12 人需求，缺口 4 人，招聘周期 3 个月
verdict：solid
理由：数据来源明确，因果链清晰。
处理：标记 ✅
```

### needs-work — 论据有漏洞

```
议题：1.2 技能不匹配
第一轮追问：
- 假设：团队缺乏前端技能
- 质疑：有技能矩阵吗？
- 回答：没有，是项目经理的主观感觉

verdict：needs-work
理由：核心前提缺乏数据支撑。
处理：补充追问——
  - 是否有因技能问题导致的返工记录？
  - 哪些任务延期了？所需技能是什么？

补充后发现：近 3 个月 5 次前端返工，涉及 React 性能优化
verdict（修订）：solid
处理：标记 ✅，caveat：建议建立正式技能矩阵
```

### invalid — 前提不成立

```
议题：3. 技术问题（假设：技术债务导致延期）
追问：
- 假设：技术债务拖慢开发
- 质疑：有量化指标吗？
- 回答：代码乱，老代码多，是开发的抱怨
- 质疑：延期的任务有多少是改老代码导致的？
- 回答：大部分延期是新增需求，不是改老代码

verdict：invalid
理由："技术债务导致延期"前提不成立，实际延期主要是新增需求。
处理：与用户确认后，议题 3 调整为"需求管理问题"，重新拆解：
  ├─ 3.1 需求变更频繁
  └─ 3.2 需求范围失控
旧议题 3 标记 ⊘（已废弃）
```

---

## 要点

- 禁止跳过 verdict 直接标记 ✅——即使显然成立也要标注 `solid` + 1 句理由
- `needs-work` 最多补充追问 2 轮，仍不足则带 caveat 标 ✅
- `invalid` 不代表白费，是发现方向性错误，避免继续投入
- verdict 必须附"理由"字段
