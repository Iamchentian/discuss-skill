# Verdict 裁决示例

深度追问后必须给出 verdict。三种 verdict 的完整示例。

## 1. solid — 论证充分

```
议题：1.1 人员不足

深度追问：
- 当前团队规模：8 人
- 项目需求：12 人（依据：项目计划书第 3 页人力需求表）
- 离职率：20%（依据：HR 季度报告）
- 招聘周期：3 个月（依据：近半年招聘数据）

verdict：solid
理由：数据来源明确，因果链清晰——8 人 < 12 人需求，缺口 4 人且短期内难以补齐。
处理：标记议题 ✅，记录论据
```

## 2. needs-work — 有漏洞

```
议题：1.2 技能不匹配

深度追问（第一轮）：
- 假设：团队缺乏前端技能
- 质疑：你怎么知道缺乏？有技能矩阵吗？
- 回答：……没有正式的技能矩阵，是项目经理的主观感觉

verdict：needs-work
理由：核心前提"技能不匹配"缺乏数据支撑，仅凭主观判断。
处理：补充追问 1-2 轮——
  - 追问：是否有近期因技能问题导致的任务返工记录？
  - 追问：哪些具体任务延期了？延期任务所需技能是什么？

补充追问后：
- 追问发现：近 3 个月有 5 次前端任务返工，涉及 React 性能优化
- 数据支撑：Jira 返工记录

verdict（修订）：solid
理由：补充了返工记录，技能短板聚焦到 React 性能优化领域。
处理：标记议题 ✅，caveat：建议后续建立正式技能矩阵
```

## 3. invalid — 前提不成立

```
议题：3. 技术问题（假设：技术债务导致延期）

深度追问：
- 假设：技术债务拖慢了开发速度
- 质疑：技术债务具体指什么？有量化指标吗？
- 回答：代码库比较乱，老代码多
- 质疑：代码乱和延期有直接因果关系吗？有没有数据？
- 回答：……没有直接数据，是开发同学的抱怨
- 质疑：那延期的任务，有多少是因为改老代码导致的？
- 回答：实际上大部分延期任务是新增需求，不是改老代码

verdict：invalid
理由："技术债务导致延期"的前提不成立——延期任务主要是新增需求，与老代码无关。方向错误。
处理：与用户确认——
  技能：议题 3"技术问题"的 verdict 为 invalid。
  追问方向"技术债务导致延期"不成立，实际延期主要是新增需求。
  建议将议题 3 调整为"需求管理问题"，重新拆解。是否同意？

用户：同意

技能：议题 3 调整为"需求管理问题"，回退到拆解阶段：
├─ 3.1 需求变更频繁
└─ 3.2 需求范围失控
旧议题 3 标记为 ❌（已废弃），记录修订事件
```

## Verdict 使用要点

- **禁止跳过 verdict 直接下结论**——即使是显然成立的议题，也要明确标注 `solid` 并附 1 句理由
- `needs-work` 不是失败，是诚实地标注论据不足；最多补充追问 2 轮，仍不足则带 caveat 标记 ✅
- `invalid` 不代表讨论白费，而是发现了方向性错误，避免在错误方向上继续投入
- verdict 必须附带"理由"字段，1-2 句话说明判断依据
